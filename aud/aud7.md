Chunk 7 Audit: ops (launch / serve / watchdog / config / template / deploy)
Functions analyzed
launch.py
- main (129-213) — Orchestrates: (1) Prefect API readiness wait with 90s budget, (2) dashboard thread, (3) FY rollover, (4) concurrency-limit provisioning, (5) orphaned-run cleanup, (6) Prefect serve() in the main thread. Logic is sound. Two distinct config loads in this file: one inside _run_dashboard (thread) at launch.py:30 and one at the bottom of main at launch.py:186 — both are lazy, both will trigger Pydantic re-validation (idempotent but wasteful; ~50-200ms each).
- _run_dashboard (27-37) — Defers models.config import and load_config into the thread, then runs uvicorn. Correct zero-import-at-module-load pattern.
- _check_prefect_api (40-51) — Single-shot health check via httpx.get on /health. Catches httpx.ConnectError explicitly and a blanket Exception — silently swallows HTTP errors. The blanket catch hides real misconfiguration (e.g. wrong base URL).
- _ensure_concurrency_limit (54-86) — Creates the global limit aam-backup (used by flow.py:concurrency("aam-backup")) and a tag-based limit. The tag-based one is created via create_concurrency_limit which raises on duplicate; the bare except Exception: pass (line 82) silently swallows every error including programming bugs. Idempotency check is brittle — relies on the API raising on duplicate; a partial-failure network drop will print a success message without actually creating the limit.
- _cancel_orphaned_runs (89-126) — Cancels PENDING and RUNNING runs in two passes. Uses force=True which is correct for stuck PENDINGs. Caveat: also cancels RUNNING runs that the Prefect worker is actively processing — could interrupt a healthy in-progress backup if launch.py is restarted mid-run (e.g. via watchdog) without the lock file being checked first. The watchdog's lock-protocol is not consulted here.
serve.py
- deployments (16-18) and _deployments (21-58) — Pointless wrapper indirection. The public deployments() does nothing but call _deployments(). The only difference is a docstring. Both are referenced from launch.py:196 (from serve import deployments) and __main__ block calls _deployments() directly (inconsistent).
- _deployments (21-58) — Creates four deployments. Config load IS deferred: load_config is called inside the function body, not at import. The from flow import backup, monthly_report_flow, weekly_report_flow at module top-level (line 12) is fine — @flow decorators don't validate config.
- __main__ block (61-63) — Calls _deployments() then serve(*d, pause_on_shutdown=False). Imports serve from prefect at module top — name-collision with the file's own name serve.py. Works because from prefect import serve binds the symbol before serve (the filename) is referenced; the implicit __main__ lookup of serve (the name) inside serve.py actually resolves to the prefect.serve function on line 62. Confusing but not broken.
watchdog.py
- _configure_logging (55-66) — Re-implements what core/logging.py:configure() already does (with slightly different rotation/retention and an extra stdout sink). Should call configure() from core.logging instead of configuring loguru directly.
- _pid_is_alive (71-80) — Uses Windows tasklist /FI "PID eq N" /NH. Reimplements core/process.py:pid_alive which uses psutil.pid_exists — the cross-platform version that's already in the codebase and already tested. Should import and use it. The string match str(pid) in result.stdout (line 78) is loose: would false-positive if a process name contained the digits (e.g. PID 12 vs. line "tasklist 12.exe").
- _is_backup_running (83-122) — Solid logic: PID-stamped lock file primary check, stale-lock cleanup, fallback to rclone.exe/robocopy.exe process detection. Correctly handles ValueError on garbage in lock file. Race window: between read_text and _pid_is_alive check the process could die; the fallback process-detection path is the safety net.
- _service_is_running (127-140) — Misnamed: it returns True only when state contains "RUNNING", so STOP_PENDING and START_PENDING both return False. The caller (line 242) treats False as "service is transitioning — wait" which is correct, but the name is misleading.
- _stop_service (143-164) — Calls sc stop with 30s timeout. Logs returncode and stderr. Reasonable.
- _check_health (169-176) — Single-shot Prefect API health check with 10s timeout. Blanket except Exception hides configuration bugs.
- main (181-265) — Main loop with sensible state machine: increments failures, defers on backup lock, force-restarts after MAX_DEFERRALS=15 deferrals (~30min) to defeat PID-reuse scenarios. Good operational design.
models/config.py
- load_config (232-234) — CRITICAL impact (per gitnexus_impact: 8 direct callers, 17 processes). Trivial wrapper over from_yaml.
- from_yaml (225-229) — Opens file, calls yaml.safe_load (PyYAML), constructs AppConfig. Inconsistency: pyproject.toml:21 lists ruamel.yaml>=0.18.0 as a dependency, but from_yaml uses PyYAML's safe_load — ruamel.yaml is only used by core/fy_rollover.py:update_config_yaml for round-trip preservation. Either (a) from_yaml should use ruamel for consistency, or (b) the ruamel dep should be dropped. The current state is dead weight in the lockfile.
- AppConfig + sub-models — Pydantic v2 patterns are correct. __repr__/__str__ redacts smtp_password (line 149) and api_key (line 181). Good security pattern. model_validator enforces api_key required when auth_enabled (line 175-178). valid_cron validator (line 195-201) checks field count only — doesn't validate range bounds, but Prefect will catch that at deployment time.
- PathsConfig.is_unc_path (31-36) — Regex ^\\\\.+\\ matches the prefix structure but doesn't verify the share name is non-empty or that there's at least one character between the server and share parts. Permissive but not broken.
templates/dashboard.py
- render_dashboard (261-349) — Pure function, zero I/O, zero imports outside stdlib. Good separation. NONE of the 17 interpolated values are html-escaped in the f-string (lines 289-348). Trust is delegated to the caller (ui.py). The downstream _render_dashboard in ui.py partially escapes (html.escape(cr['error'][:60]) on lines 579, 584) but does NOT escape cr['status'] when building cloud_run / lan_run strings, and does NOT escape r["mode"] or r.get("error_message", "")[:60] when building history_rows (line 625, 626). See Security section.
- Inline JS constant (62-230) — updateStatus and triggerBackup handlers. The escapeHtml closure defined inline (line 194) is correct, and IS applied to r.error, r.started_at, r.status, r.files, r.duration (line 221). However modeTag (line 195) escapes r.mode.toUpperCase() but does NOT validate the r.mode value before splicing it into a CSS class: <span class="tag ' + r.mode + '">. A db record with mode = '" onerror="alert(1)' would break out of the class attribute.
- REPORT_JS (232-258) — Clean.
deploy/download_nssm.py
- Whole file is a single top-level script (no functions). Flow: download NSSM Chocolatey nupkg → extract → find inner tools/*.zip → extract → glob win64/nssm.exe → copy to C:\BackupAgent\tools\nssm.exe. No hash verification, no signature check, no file-size sanity check on the downloaded binary. The package is fetched from a public Chocolatey CDN over HTTPS (TLS verified by urllib default), but integrity is not validated. The binary will be invoked with Administrator privileges by the PowerShell wrapper (download_nssm.ps1:5 #Requires -RunAsAdministrator). Supply-chain risk.
- Line 28: Spoofs User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64). Cosmetic anti-pattern — the urllib default UA is honest; this adds a false identity.
- Line 51: import glob is mid-function (line 51). Style nit — should be at top.
- Line 73: nssm_src = nssm_src_candidates[0] picks the first glob match. If a future NSSM release ships both win64 and win32 binaries in the inner zip, this picks whichever is first alphabetically — not deterministic enough.
- Line 11-12: os.environ.get("TEMP", r"C:\Temp") — on non-Windows TEMP is set to /tmp so the path becomes POSIX, but C:\BackupAgent\tools (line 9) is still hardcoded Windows. Module will crash on non-Windows at the os.makedirs call.

Anti-patterns
- watchdog.py:55-66 _configure_logging — Re-implements core/logging.py:14-40 (file rotation, loguru remove() + add() pattern, format string). Should from core.logging import configure as configure_logging; configure_logging(LOG_DIR).
- watchdog.py:71-80 _pid_is_alive — Re-implements core/process.py:pid_alive. Loses cross-platform support for free.
- watchdog.py:78 str(pid) in result.stdout — Substring match. Vulnerable to a malicious-named process containing the PID digits. Use regex \bpid\b or parse tasklist table.
- watchdog.py:82, 119, 139, 164 bare except Exception: pass — Swallows real errors (typos in service name, permission denied, network drops). At minimum, log the exception.
- watchdog.py:127-140 _service_is_running — Misleading name; returns False for START_PENDING / STOP_PENDING. Rename to _service_is_in_running_state or have the caller check explicitly.
- serve.py:16-18 deployments() wrapper — Pure delegation, adds a public alias for no reason.
- serve.py:13 from models.config import CONFIG_PATH, load_config at top-level — Imports load_config at import time but only uses it inside _deployments(). The CONFIG_PATH constant import is necessary (used in deployment parameters). Move load_config to a lazy import inside the function to keep the "deferred config" promise the docstring claims.
- launch.py:22 os.environ["PREFECT_API_URL"] = "http://127.0.0.1:4200/api" at module top-level — Mutates global env at import. Any code that imports launch (even for its name) gets a side effect. Move into main().
- launch.py:186-187 load_config re-called in main() after _run_dashboard already loaded it — Wasteful, but the Pydantic models are immutable so semantically fine.
- launch.py:89-126 _cancel_orphaned_runs — Cancels RUNNING runs (line 101) without consulting backup.lock. If a backup is actively running, cancelling its Prefect flow run will mark the work as failed even though the lock-protected subprocess is still writing data.
- launch.py:142-150 for _elapsed in range(0, _API_MAX_WAIT, _API_INTERVAL) — Uses a for loop as a timer. The _remaining = _API_MAX_WAIT - _elapsed - _API_INTERVAL math is brittle (last iteration computes negative, hence the max(0, _remaining)). A while with time.monotonic() would be clearer.
- models/config.py:9 import yaml (PyYAML) but pyproject.toml:21 lists ruamel.yaml — Two YAML libs in the project for different purposes. from_yaml does NOT preserve comments; only core/fy_rollover.py:update_config_yaml does. Inconsistent. Pick one.
- models/config.py:232-234 load_config — One-line wrapper, no added value. Callers could use AppConfig.from_yaml(p) directly.
- templates/dashboard.py:289-348 — F-string HTML with 17 unescaped interpolations. Architectural anti-pattern: pushes the escaping responsibility onto every caller. ui.py calls it and partially escapes — guarantees eventual drift.
- templates/dashboard.py:195 '<span class="tag ' + r.mode + '">' + escapeHtml(r.mode.toUpperCase()) + '</span>' — r.mode is spliced into the class attribute unescaped. If a db row has mode = '" onclick="alert(1)" data-x="', it would break out. (Mitigated in practice because mode is constrained to "cloud" / "lan" by flow.py, but the template shouldn't assume this.)
- deploy/download_nssm.py:8 chocolatey_url = "https://community.chocolatey.org/api/v2/package/nssm" — HTTPS ✓, but no integrity pin. NSSM Chocolatey package version not pinned either — every install fetches "latest".
- deploy/download_nssm.py:73 nssm_src = nssm_src_candidates[0] — First-match is not deterministic across glob.glob order. Should prefer win64 explicitly, error if multiple.
- deploy/download_nssm.py:51 import glob mid-function — Style nit.
- launch.py:1-13 docstring — Says "Dashboard UI (in-process, configurable host:port)" and "Backup scheduler (in-process, 4 deployments)" but does not mention FY rollover, concurrency limits, or orphaned-run cancellation that the body performs. Stale.
Security
Severity	Location	Finding
HIGH	deploy/download_nssm.py:24-35	No hash/signature verification of the downloaded nssm.exe. Binary is installed to C:\BackupAgent\tools\nssm.exe and runs as a Windows service controller (full system access). If Chocolatey's CDN or a transitive mirror is compromised, an attacker gets SYSTEM on the backup host. Mitigation: pin to a known version + SHA-256 from NSSM's official source (https://nssm.cc/release (https://nssm.cc/release)).
HIGH	templates/dashboard.py:289-348 + ui.py:610-626	Stored XSS in the dashboard's history table. ui.py:611 interpolates r["mode"] into both a CSS class attribute AND as uppercased text — neither escaped. ui.py:625 interpolates r.get("error_message", "")[:60] without html.escape(). An attacker with write access to the SQLite runs table (or anyone who can plant a row via a compromised dependency) can execute JS in the dashboard. Cross-platform issues
The project is Windows-only end-to-end (start.bat, NSSM, shutdown.exe, tasklist, sc, UNC paths, C:\BackupAgent). pyproject.toml:6 declares requires-python = ">=3.12" with no platform marker, and there is no [project] classifiers = ["Operating System :: Microsoft Windows"] hint.
File	Non-Windows behaviour
watchdog.py:74-80, 108-118, 134-140, 153-160	subprocess.run(["tasklist", ...]) raises FileNotFoundError on Linux/macOS. Caught by bare except Exception: pass → watchdog silently returns "no backup / not running" → triggers restart of a service that doesn't exist. Fail-unsafe.
core/shutdown.py:24-33	shutdown command not on PATH on non-Windows. FileNotFoundError is caught and a warning logged — graceful.
deploy/download_nssm.py:9-13, 76	Hardcoded C:\BackupAgent\tools and C:\Temp paths. os.makedirs on C:\... from Linux fails with FileNotFoundError or wrong path.

flow.py:604-613, 663	Lock-file write uses Path.parent (POSIX-safe). os.getpid() is cross-platform. No cross-platform issue here.
models/config.py + all Pydantic models	Pure validation, no platform-specific code. No issue.
