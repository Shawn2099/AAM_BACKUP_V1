# AAM Backup Automation V1 — System Dictionary

*This document serves as the "brain" for AI agents and new developers, summarizing the exact purpose, inputs, and architectural constraints of every module in the codebase. Read this file instead of re-reading the entire codebase.*

## Core Architectural Rules
1. **Config-Driven**: `config.yaml` (validated by `models/config.py`) is the absolute source of truth. No hardcoded paths, timeouts, or retries are allowed in Python.
2. **HDD-Optimized**: The source drive is a slow mechanical HDD. Parallel I/O causes thrashing. Rclone/robocopy concurrency is strictly throttled, and unnecessary hashing/reads are banned.
3. **Dual Destination**:
   - **LAN (`robocopy`)**: 1:1 Mirror for instant Business Continuity/DR. Never rolls over.
   - **Cloud (`rclone`)**: Segmented by Fiscal Year (e.g. `FY26-27`) for long-term compliance and off-site disaster protection.
4. **Autonomous Rollover**: The system gracefully handles the April 1st fiscal year boundary without human intervention.

## Root Orchestrators

### `launch.py`
**Purpose:** The primary entry point for nightly backups.
**Behavior:** Instantiates the config, triggers `core/fy_rollover.py` to check if a Fiscal Year boundary was crossed, and then executes the Prefect flows located in `flow.py`.

### `flow.py`
**Purpose:** Prefect 3.x Orchestration.
**Behavior:** Ties all isolated `core/` modules together into execution DAGs (Directed Acyclic Graphs).
**Key Flows:** 
- `run_nightly_backup()`: The master flow that runs preflight -> sync -> verify -> report.
- Distinct task functions for Cloud and LAN isolation.

### `ui.py`
**Purpose:** FastAPI dashboard.
**Behavior:** Provides a lightweight web interface to trigger manual backups, view recent logs, and download HTML reports generated by `core/report.py`.

### `watchdog.py`
**Purpose:** Process supervisor.
**Behavior:** Ensures that if the backup process hangs catastrophically, it is terminated. Prevents multiple backup instances from running simultaneously.

## Core Modules (`core/`)

### 1. Synchronization & Data Movement
- **`cloud_sync.py`**: Executes `rclone sync` with GCS-optimized flags. Appends the Fiscal Year prefix to the destination.
- **`lan_sync.py`**: Executes `robocopy /MIR`. Parses 25-year-old robocopy exit codes (0-7 are success/partial, >8 is failure) to determine sync health.
- **`rclone_config.py`**: Dynamically generates a temporary `rclone.conf` file containing GCS service account credentials so they aren't stored globally on the server.

### 2. Preflight & Health Checks
- **`health.py`**: The gatekeeper. Checks for critical free space on the source drive, ensures `rclone`/`robocopy` binaries exist, and verifies that the Windows clock skew isn't >10 minutes (which breaks GCS OAuth).
- **`cloud_preflight.py`**: Uses a "Two-Probe" architecture (local stat check + fast GCS bucket check) to verify network and credentials without doing a heavy HDD read.
- **`lan_preflight.py`**: Checks if the NAS is mounted and accessible before syncing. *(Planned: Canary File `.AAM_TARGET_MOUNTED` check to prevent phantom mount data deletion).*

### 3. Verification & Reporting
- **`cloud_verify.py`**: Runs `rclone check` post-sync. *(Planned: Use `--size-only` to prevent 2-hour HDD re-hashing).*
- **`report.py`**: Reads `ManifestDB` to generate HTML summaries of backup success/failure. Sends SMTP email alerts on failure and weekly/monthly summaries.
- **`cloud_reporter.py`**: Runs read-only `rclone` commands (`size`, `lsjson`, `check --combined`) to pull actual state from GCS.

### 4. Fiscal Year Rollover
- **`fy_rollover.py`**: The April 1st logic. 
  1. Detects boundary crossing.
  2. Runs one final backup to the old FY.
  3. Transitions old GCS data to cold `ARCHIVE` storage to save money.
  4. Auto-creates the new year folders (`D:\FY27-28`).
  5. Atomically rewrites `config.yaml` to point to the new folder.

### 5. Infrastructure & State
- **`manifest.py`**: `ManifestDB`. A SQLite WAL-mode database that tracks backup histories, bytes transferred, and exit codes. 
- **`time_utils.py`**: Standardized `pendulum` wrapper for calculating IST timestamps and the `FY26-27` string.
- **`wol.py`**: Wake-on-LAN. Sends Magic Packets to wake the physical LAN NAS from sleep before `lan_sync.py` runs.
- **`shutdown.py`**: Sends `shutdown.exe` commands to power down the NAS after the backup is complete to save electricity and isolate it from ransomware.
- **`process.py`**: Safely wraps `subprocess` and resolves binary paths (like checking if `rclone.exe` is in the system PATH).

## Data Models (`models/config.py`)
The system strictly parses `config.yaml` into Pydantic v2 models. Key models:
- **`PathsConfig`**: `source_drive`, `lan_destination`, `database_path`, `gcs_key_path`.
- **`LanConfig`**: Retries, timeouts, multi-threading (`mt_threads`).
- **`CloudConfig`**: GCS-specific settings, bandwidth limits, buffer size.
- **`WolConfig`**: MAC address, IP, and ping intervals for Wake-on-LAN.
- **`MaintenanceConfig`**: SQLite tuning (e.g., `sqlite_vacuum_freelist_threshold`).

## Database Schema (`ManifestDB`)
Located in `core/manifest.py`. Uses SQLite in WAL (Write-Ahead Log) mode.
1. **`file_entries` table**: Tracks every file (`relative_path`, `file_size`, `mtime`, `md5_checksum`, `lan_status`, `cloud_status`).
2. **`run_history` table**: Tracks every Prefect execution (`run_id`, `mode`, `started_at`, `status`, `exit_code`, `files_copied`, `bytes_copied`).

## Execution Constraints & CLI Flags
The Python code generates highly optimized CLI strings to protect the mechanical HDD.

**1. Rclone (Cloud Sync)**
```bash
rclone sync D:\ aam_gcs:bucket/FY26-27 --fast-list --gcs-no-check-bucket --error-on-no-transfer \
  --transfers 2 --checkers 4 \
  --modify-window 2s --check-first \
  --buffer-size 64M
```
* **`--transfers 2 --checkers 4`**: Throttled to prevent HDD thrashing.
* **`--modify-window 2s`**: Matches NTFS 2-second timestamp granularity — prevents false re-uploads.
* **`--check-first`**: Forces metadata comparisons to finish before uploads start, cleanly separating random-seek I/O from sequential-read I/O.
* **`--error-on-no-transfer`**: Returns exit 9 when nothing changed (distinguishable from exit 0).
* **Exit Codes**: `0` = `CLOUD_COMPLETE` (files synced), `4` = `CLOUD_PARTIAL` (transferred with errors), `9` = `CLOUD_NO_CHANGES_COMPLETE` (nothing to sync).

**2. Robocopy (LAN Sync)**
```bash
robocopy D:\ \\192.168.10.10\share$ /MIR /Z /ZB /XJ /MT:4 /R:3 /W:10 /V /TS /FP /NJH /NJS /NDL /NP
```
* **`/MIR`**: Mirrors source to destination (purging extras).
* **`/MT:4`**: Limits multi-threading to 4 to protect the HDD read head.
* **`/Z /ZB`**: Restartable mode in case the LAN drops.
* **Exit Codes**: Bitmask mapping. `0-3` = `LAN_COMPLETE` (Success). `4-7` = `LAN_PARTIAL` (Mismatches/Extras). `8+` = `LAN_FAILED` (Fatal errors).

## Execution Sequence (`flow.py`)
1. `validate_preconditions`: Checks disk space, binary paths, clock skew (`core/health.py`).
2. `wake_lan_target`: Sends Magic Packet (`core/wol.py`).
3. `lan_preflight` / `cloud_preflight`: Checks network reachability via Two-Probe method.
4. `lan_sync_task` / `cloud_sync_task`: Executes the CLI binaries.
5. `verify_cloud_integrity_task`: Runs `rclone check`.
6. `shutdown_lan_target`: Sends WoL shutdown command.
7. `generate_and_send_report`: Compiles `ManifestDB` stats into HTML and emails the admin.
